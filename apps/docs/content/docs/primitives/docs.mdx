## Purpose

`tx doc` manages structured documentation as first-class primitives. Docs are YAML files on disk (`.tx/docs/`) with metadata and links stored in the database. This enables:

- **Version-controlled docs** that live alongside code
- **Linked doc graphs** connecting overviews, PRDs, and design docs
- **Rendered markdown** generated from YAML source
- **Invariant extraction** from design docs for machine-checkable rules

## EARS in tx PRDs

tx now supports **EARS** (**Easy Approach to Requirements Syntax**) for PRD requirements through the optional `ears_requirements` block.

Why this is now used:

- Plain bullet requirements are easy to write but can be ambiguous.
- EARS gives requirements a predictable structure that is easier to lint and review.
- EARS IDs map cleanly to test cases in DD testing strategies (traceability).
- tx can validate EARS early with `tx doc lint-ears` before implementation starts.

EARS in tx uses these patterns:

| Pattern | Meaning | Required extra field(s) |
|------|---------|--------------------------|
| `ubiquitous` | Always true behavior | none |
| `event_driven` | Triggered by an event | `trigger` |
| `state_driven` | Valid while a state holds | `state` |
| `optional` | Conditional by feature/config | `feature` |
| `unwanted` | Protective/exception behavior | `condition` |
| `complex` | Combination of clauses | at least one of `trigger`/`state`/`feature`/`condition` |

Use deterministic IDs: `EARS-<AREA>-NNN` (example: `EARS-AUTH-001`).

## Doc Kinds

| Kind | Purpose |
|------|---------|
| `overview` | Project-level overview (one per project) |
| `prd` | Product Requirements Document |
| `design` | Design Document (technical specification) |

## Lifecycle

Docs have two statuses:

```
changing → locked
```

- **changing** — editable, can be updated freely
- **locked** — immutable, must create a new version to make changes

## Usage

<Tabs groupId="interface" persist items={["CLI", "TypeScript SDK", "MCP", "REST API"]}>
<Tab value="CLI">

```bash
# Create a new doc
tx doc add prd auth-system --title "Authentication System"

# Edit a doc (opens YAML in $EDITOR)
tx doc edit auth-system

# Show doc metadata
tx doc show auth-system

# List all docs
tx doc list
tx doc list --kind prd
tx doc list --status locked

# Render docs to markdown
tx doc render
tx doc render auth-system

# Lock a doc (makes it immutable)
tx doc lock auth-system

# Create a new version (doc must be locked first)
tx doc version auth-system

# Link two docs
tx doc link overview auth-system

# Attach a task to a doc
tx doc attach tx-abc123 auth-system

# Create a design patch
tx doc patch auth-design auth-patch-001 "Fix auth flow"

# Validate doc graph
tx doc validate

# Detect drift between YAML and DB
tx doc drift auth-system

# Validate EARS requirements for a PRD (by doc name or YAML file path)
tx doc lint-ears auth-system
tx doc lint-ears .tx/docs/prd/auth-system.yml
```

<Tab value="TypeScript SDK">

```typescript
import { TxClient } from '@jamesaphoenix/tx-agent-sdk'

const tx = new TxClient({ apiUrl: 'http://localhost:3456' })

// List all docs
const docs = await tx.docs.list()

// List PRDs only
const prds = await tx.docs.list({ kind: 'prd' })

// Get a doc by name
const doc = await tx.docs.get('auth-system')

const created = await tx.docs.create({
  kind: 'prd',
  name: 'auth-system',
  title: 'Authentication System',
  yamlContent: 'kind: prd\ntitle: Authentication System\n...',
})

// Update YAML content
const updated = await tx.docs.update('auth-system', newYamlContent)

// Lock a doc (makes it immutable)
const locked = await tx.docs.lock('auth-system')

// Link two docs
const link = await tx.docs.link('overview', 'auth-system')

// Render to markdown
const rendered = await tx.docs.render('auth-system')

// Get the doc graph
const graph = await tx.docs.graph()
```

## YAML Structure

Docs are stored as YAML files in `.tx/docs/`:

```
.tx/docs/
  overview.yml          # Project overview
  prd/
    auth-system.yml     # PRD
    data-pipeline.yml
  design/
    auth-design.yml     # Design doc
    auth-patch-001.yml  # Design patch
  index.yml             # Auto-generated index
  index.md              # Auto-generated markdown index
```

Example PRD YAML:

```yaml
kind: prd
name: auth-system
title: Authentication System
status: changing
version: 1

problem: Users cannot log in to the application
solution: Implement JWT-based authentication

requirements:
  - User registration with email/password
  - Login with JWT tokens
  - Password reset flow

ears_requirements:
  - id: EARS-AUTH-001
    pattern: event_driven
    trigger: a user submits valid credentials
    system: authentication service
    response: issue access and refresh tokens
    priority: must
    rationale: core login success path
    test_hint: test/integration/auth-login.test.ts

acceptance_criteria:
  - Users can register and log in
  - Tokens expire after 15 minutes
  - Refresh token rotation works

invariants:
  - id: INV-AUTH-001
    rule: Passwords must be hashed with bcrypt
    enforcement: integration_test
    test_ref: test/auth.test.ts
```

`requirements` and `ears_requirements` can coexist. This keeps backwards compatibility while allowing structured, test-traceable requirements.

## Doc Graph

Docs form a directed acyclic graph (DAG):

```
overview ──overview_to_prd──► prd ──prd_to_design──► design
                                                        │
                                               design_patch
                                                        │
                                                        ▼
                                                   design-patch
```

Link types are auto-inferred from doc kinds when not specified explicitly.
