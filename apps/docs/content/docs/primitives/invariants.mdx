## Purpose

Invariants are machine-checkable rules extracted from design documents. They bridge the gap between documentation and enforcement. Each invariant has a defined enforcement mechanism (integration test, linter, or LLM-as-judge) and an audit trail of check results.

## Enforcement Types

| Type | Description | Example |
|------|-------------|---------|
| `integration_test` | Verified by running a test | `test/auth.test.ts` |
| `linter` | Verified by a lint rule | `tx/require-taskwithdeps-return` |
| `llm_as_judge` | Verified by LLM evaluation | `prompts/check-auth.md` |

## Invariant IDs

Invariants use a structured ID format: `INV-[A-Z0-9-]+`

Examples: `INV-AUTH-001`, `INV-DB-SCHEMA-002`, `INV-API-RESPONSE-003`

## Audit Trail

Every invariant check is recorded with:

- Whether it passed or failed
- Optional details (error messages, test output)
- Timestamp and duration

This creates an audit trail showing when invariants were last verified and their pass/fail history.

```bash
# List all invariants
tx invariant list
tx invariant list --subsystem auth

tx invariant show INV-AUTH-001

tx invariant record INV-AUTH-001 --passed

tx invariant sync
tx invariant sync --doc auth-design
```

## Defining Invariants in YAML

Invariants are defined in the `invariants` section of a design doc YAML:

```yaml
kind: design
name: auth-design
title: Authentication Design

invariants:
  - id: INV-AUTH-001
    rule: Passwords must be hashed with bcrypt before storage
    enforcement: integration_test
    subsystem: auth
    test_ref: test/auth.test.ts

  - id: INV-AUTH-002
    rule: JWT tokens must expire within 15 minutes
    enforcement: integration_test
    subsystem: auth
    test_ref: test/jwt.test.ts

  - id: INV-AUTH-003
    rule: Login endpoints must be rate-limited
    enforcement: linter
    subsystem: auth
    lint_rule: security/rate-limit-required

  - id: INV-AUTH-004
    rule: Auth error messages must not reveal user existence
    enforcement: llm_as_judge
    subsystem: auth
    prompt_ref: prompts/check-auth-errors.md
```

## Syncing Invariants

Running `tx invariant sync` reads invariant sources from each doc's YAML and upserts them into the database:

- explicit `invariants` arrays,
- PRD `ears_requirements` (EARS-structured behavioral requirements),
- PRD `requirements` (plain-text, auto-derived as `INV-PRD-<docname>-REQ-NNN`),
- design `goals` (auto-derived as `INV-DESIGN-<docname>-GOAL-NNN`).

For docs that use block-scalar strings for `requirements` or `goals`, sync normalizes non-empty lines into invariant candidates.

Invariants that are no longer present in YAML are marked as `deprecated`.

```bash
# Sync all docs
tx invariant sync

# Sync a specific doc
tx invariant sync --doc auth-design
```

## Invariant Provenance

Every invariant tracks where it came from via `source` and `source_ref` fields:

| Source | Description | Example |
|--------|-------------|---------|
| `explicit` | Manually authored in YAML `invariants[]` | Default for all hand-written invariants |
| `goals` | Auto-derived from design doc `goals[]` | `INV-DESIGN-AUTH-GOAL-001` |
| `decision` | Generated from an approved decision | `source_ref: dec-abc123def456` |
| `constraint` | System design constraint | `INV-SD-ERROR-001` |

## EARS Fields on Invariants

Invariants derived from EARS-structured requirements carry full EARS metadata in the database:

| Field | Description |
|-------|-------------|
| `pattern` | EARS pattern (ubiquitous, event_driven, state_driven, unwanted, optional, complex) |
| `system_name` | The system/subject (e.g., "authentication service") |
| `response` | The SHALL behavior |
| `trigger_text` | WHEN clause (event_driven) |
| `state_text` | WHILE clause (state_driven) |
| `condition_text` | IF clause (unwanted) |
| `feature` | WHERE clause (optional) |
| `rationale` | Why the invariant exists |
| `test_hint` | Suggested test file/approach |

The `rule` field is auto-computed from EARS components using `renderEarsRule()`.

## Invariant Statuses

| Status | Description |
|--------|-------------|
| `active` | Currently enforced |
| `deprecated` | No longer in YAML, kept for audit trail |
