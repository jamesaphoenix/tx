## 1. Compiled Binary Version Crash (CRITICAL)

**Problem**: `bun build --compile` creates binaries with a virtual `/$bunfs/` filesystem. The `version.ts` file used `readFileSync` to read `package.json` at runtime, but the minifier hoisted it out of the try-catch block to module scope. In compiled binaries, `/$bunfs/package.json` doesn't exist, causing an unrecoverable ENOENT crash.

**Root cause**: Bun's minifier optimization moves `readFileSync` calls to module scope when it detects they're used in a simple pattern, ignoring the try-catch that was supposed to handle the error.

**Fix** (`apps/cli/src/version.ts`):

```typescript
// BEFORE — minifier hoists readFileSync out of try-catch
import { readFileSync } from "node:fs"
try {
  const pkg = JSON.parse(readFileSync("../package.json", "utf-8"))
  return pkg.version
} catch { return "0.0.0" }

// AFTER — require() prevents hoisting + compile-time injection
const SEMVER_RE = /^\d+\.\d+\.\d+(-[a-zA-Z0-9._-]+)?(\+[a-zA-Z0-9._-]+)?$/

function getVersion(): string {
  // Compile-time injection — bun replaces this with a literal string
  const injected = process.env.TX_CLI_VERSION
  if (injected && injected !== "undefined" && injected !== "" && SEMVER_RE.test(injected)) {
    return injected
  }

  // Development fallback using require() to prevent minifier hoisting
  try {
    const { readFileSync } = require("node:fs") as typeof import("node:fs")
    const { resolve, dirname } = require("node:path") as typeof import("node:path")
    const { fileURLToPath } = require("node:url") as typeof import("node:url")
    const __dir = dirname(fileURLToPath(import.meta.url))
    const pkg = JSON.parse(readFileSync(resolve(__dir, "../package.json"), "utf-8"))
    return pkg.version ?? "0.0.0"
  } catch { return "0.0.0" }
}
```

**Key learnings**:
- `require()` inside a function body is NOT hoisted by Bun's minifier — `import` at module scope IS
- `--define "process.env.TX_CLI_VERSION='$VERSION'"` injects a string literal at compile time
- The compile-time check short-circuits before any filesystem access in production binaries
- End-anchored semver regex (`$`) prevents accepting strings like `1.2.3; malicious`

## 2. Embedded Migrations for Compiled Binaries (CRITICAL)

**Problem**: Compiled binaries can't read `migrations/*.sql` from disk at runtime because the migration directory isn't bundled into the `/$bunfs/` virtual filesystem.

**Fix**: A build-time script (`scripts/bundle-migrations.js`) reads all SQL files and generates `packages/core/src/migrations-embedded.ts` with the SQL inlined as string literals.

```typescript
// Auto-generated by scripts/bundle-migrations.js — DO NOT EDIT.
export const EMBEDDED_MIGRATIONS = [
  { version: 1, description: "initial-schema", sql: `CREATE TABLE IF NOT EXISTS tasks (...)` },
  { version: 2, description: "add-indexes", sql: `CREATE INDEX IF NOT EXISTS ...` },
  // ... all 28 migrations
] as const satisfies ReadonlyArray<{ version: number; description: string; sql: string }>
```

**Runtime fallback chain** (`packages/core/src/services/migration-service.ts`):

```typescript
// Top-level await: try filesystem first, fall back to embedded
export const MIGRATIONS: readonly Migration[] = await loadMigrationsFromDir()
  .then(m => m.length >= EMBEDDED_MIGRATIONS.length ? m : EMBEDDED_MIGRATIONS)
  .catch(() => EMBEDDED_MIGRATIONS)
```

**Build pipeline integration** (`packages/core/package.json`):

```json
{
  "scripts": {
    "prebuild": "node ../../scripts/bundle-migrations.js",
    "build": "tsc -b && { rm -r ./migrations 2>/dev/null; cp -r ../../migrations ./migrations; }"
  }
}
```

**Key learnings**:
- `prebuild` lifecycle hook runs automatically before `build` in both npm and bun
- Turbo.json must include `../../migrations/**/*.sql` as build input for cache invalidation
- The fallback uses `>=` comparison (not `===`) so new filesystem migrations are picked up during development
- Integration tests verify byte-identical SQL between embedded and filesystem versions

## 3. GitHub Actions Script Injection (HIGH)

**Problem**: Using `${{ steps.*.outputs.* }}` directly in `run:` blocks allows script injection if the value contains shell metacharacters.

```yaml
# BEFORE — inline interpolation (injection risk)
run: |
  strings ./${{ matrix.artifact }} | grep -qF "${{ steps.version.outputs.value }}"
```

```yaml
# AFTER — env: block isolates from shell context
env:
  TX_CLI_VERSION: ${{ steps.version.outputs.value }}
  BUILD_ARTIFACT: ${{ matrix.artifact }}
  BUILD_TARGET: ${{ matrix.target }}
run: |
  strings "./$BUILD_ARTIFACT" | grep -qF "$TX_CLI_VERSION"
```

**Key learning**: GitHub Actions `${{ }}` expressions undergo zero shell escaping when placed in `run:` blocks. The `env:` block creates a proper shell variable that's subject to normal quoting rules. This is a well-known OWASP-class vulnerability in GitHub Actions.

## 4. npm Tarball Test File Leaks (MEDIUM)

**Problem**: Compiled test files (`*.test.js`, `*.test.d.ts`) were being included in npm tarballs for `api-server`, `agent-sdk`, and `test-utils`.

**Fix**: Add negation patterns to the `files` field in `package.json`:

```json
// api-server
"files": [
  "dist",
  "!dist/__tests__/**",
  "!dist/**/*.test.*"
]

// agent-sdk
"files": [
  "dist",
  "!dist/**/*.test.*"
]

// test-utils
"files": [
  "dist",
  "!dist/**/*.test.*",
  "README.md"
]
```

**Key learnings**:
- Directory exclusion needs `/**` suffix: `!dist/__tests__` is unreliable, `!dist/__tests__/**` works
- `npm pack --dry-run` verifies what will be included without actually creating the tarball
- The `files` field in package.json acts as a whitelist — negation patterns subtract from included globs

## 5. install.sh Standalone Binary Installer (NEW)

**Created**: `install.sh` — POSIX sh compatible installer for standalone binaries.

```bash
#!/bin/sh
# Usage: curl -fsSL https://raw.githubusercontent.com/jamesaphoenix/tx/main/install.sh | sh

set -e

main() {
  # Prerequisites
  if ! command -v curl >/dev/null 2>&1; then
    echo "Error: curl is required but not found." >&2
    exit 1
  fi

  # Platform detection
  OS=$(uname -s | tr '[:upper:]' '[:lower:]')
  ARCH=$(uname -m)
  case "$ARCH" in
    arm64|aarch64) ARCH="arm64" ;;
    x86_64|amd64)  ARCH="x64"  ;;
  esac

  # Version validation — digits and dots only
  case "$VERSION" in
    *[!0-9.]*)
      echo "Error: version '$VERSION' contains invalid characters." >&2
      exit 1 ;;
    [0-9]*.[0-9]*.[0-9]*) ;;
    *)
      echo "Error: '$VERSION' does not look like a valid version." >&2
      exit 1 ;;
  esac

  # Download and install
  ARTIFACT="tx-${OS}-${ARCH}"
  URL="https://github.com/${REPO}/releases/download/v${VERSION}/${ARTIFACT}"
  HTTP_CODE=$(curl -fSL -w '%{http_code}' -o "$TMPFILE" "$URL" 2>/dev/null) || true
  # ...
}
```

**Key learnings**:
- POSIX `case` patterns provide shell-safe validation without regex: `*[!0-9.]*` rejects any non-digit/dot character
- The two-stage validation (character check + pattern check) catches both injection attempts AND incomplete versions
- `command -v curl` is POSIX-compliant; `which curl` is not guaranteed
- `#!/bin/sh` shebang ensures compatibility with dash, ash, and other minimal shells

## 6. Binary Build Flag Consistency (MEDIUM)

**Problem**: `release.yml` had `--external @anthropic-ai/claude-agent-sdk` but `apps/cli/package.json` build scripts didn't. Local binary builds would try to bundle the claude-agent-sdk (which is dynamically imported in core).

**Fix**: Synced all 5 build scripts in `apps/cli/package.json`:

```json
"build:binary": "bun build ./src/cli.ts --compile --outfile dist/tx --minify --external @anthropic-ai/sdk --external @anthropic-ai/claude-agent-sdk --external openai --external node-llama-cpp --external @node-llama-cpp/*"
```

**Key learning**: `--external` in `bun build` tells the bundler to leave the import as-is (not bundle it). For optional peer dependencies that use `import()` dynamically, they MUST be externalized or the bundler will try to resolve them at build time and fail.

## 7. Version String Verification for All Targets (MEDIUM)

**Problem**: The `strings` binary verification step in `release.yml` only ran for some targets. Cross-compiled binaries (e.g., building linux-arm64 on ubuntu-latest) can't be executed, but `strings` can verify the version literal is embedded.

**Fix**: Removed the conditional and use `strings` for all 4 targets:

```yaml
- name: Verify binary version
  working-directory: apps/cli
  env:
    TX_CLI_VERSION: ${{ steps.version.outputs.value }}
    BUILD_ARTIFACT: ${{ matrix.artifact }}
  run: |
    strings "./$BUILD_ARTIFACT" | grep -qF "$TX_CLI_VERSION" || {
      echo "::error::Version string '$TX_CLI_VERSION' not found in binary $BUILD_ARTIFACT"
      exit 1
    }
```

**Key learning**: `strings` extracts printable character sequences from any binary — it doesn't need to execute the binary. This works for cross-compiled targets where you can't run the binary directly.
